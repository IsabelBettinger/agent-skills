# Design & User Experience Skill

This skill guides product designers through the complete design lifecycle, emphasizing early involvement in strategy phases and seamless integration with product, engineering, and quality teams.

---

## When Design Engages

**Involvement Timeline:**
- **Phase 1-2 (Weeks 1-4):** Strategy & Planning - Design participates in problem definition and feasibility assessment
- **Phase 3-4 (Weeks 5-8):** Concept & Architecture - Design leads UX exploration and creates design systems
- **Phase 5-6 (Weeks 9-12):** Build & Refinement - Design provides guidance and validates implementation
- **Phase 7 (Weeks 12+):** QA & Launch - Design validates against specifications and supports post-launch

---

## Top Design Responsibilities

1. **Problem Definition & Research** - Conduct user research, competitive analysis, and problem discovery to inform design direction
2. **Interaction Design & User Flows** - Create user journeys, wireframes, and interaction patterns that solve core problems
3. **Visual Design & Design Systems** - Develop typography, color, component libraries, and design tokens for consistency
4. **Prototyping & Validation** - Build high-fidelity prototypes and conduct user testing to validate design decisions
5. **Design Specification & Handoff** - Create detailed design specs with measurements, states, animations, and developer handoff documentation
6. **Accessibility & Inclusive Design** - Ensure WCAG AA compliance, test with assistive technologies, and design for diverse users
7. **Design System Maintenance** - Evolve design tokens, component definitions, and patterns as product evolves

---

## Best Practices: DOs and DON'Ts

### DOs

1. **Start with research before pixels** - Validate assumptions through user interviews, surveys, and competitive research before sketching
2. **Involve developers early** - Share design constraints, technical feasibility questions, and implementation considerations during design phase
3. **Use design systems from day one** - Establish component libraries, tokens, and patterns early to maintain consistency and speed handoff
4. **Document design decisions explicitly** - Record the "why" behind design choices, edge cases, and state variations to guide development
5. **Test with real users iteratively** - Conduct usability testing at concept, prototype, and beta stages to catch issues early
6. **Design for accessibility as primary** - Integrate WCAG AA requirements, keyboard navigation, and screen reader testing into every design
7. **Create comprehensive spec artifacts** - Generate design documentation that includes responsive layouts, interaction states, and animation specs

### DON'Ts

1. **Don't hand off without context** - Never pass designs to developers without recorded design rationale, component states, and edge cases
2. **Don't design in a silo** - Avoid making design decisions without input from PM, engineering, and QA perspectives on feasibility
3. **Don't create pixel-perfect mockups without validation** - Don't assume designs work without user testing; iterate based on feedback
4. **Don't ignore performance implications** - Don't design heavy interactions without considering mobile performance and loading states
5. **Don't forget mobile-first** - Don't design desktop-first and retrofit for mobile; start with constraints and scale up
6. **Don't over-design edge cases** - Avoid adding complexity to the 80% use case to handle rare edge cases; document edge case guidance instead
7. **Don't skip WCAG compliance review** - Don't assume design is accessible; conduct formal accessibility audits and testing

## Key Design Tools & Artifacts

1. **Figma / Adobe XD** - Primary design and prototyping platform; source of truth for design specs
2. **User Research Platform** (UserTesting, Maze, Validately) - Validate concepts, test prototypes, gather quantitative feedback
3. **Design System Tools** (Storybook, Zeroheight) - Document and maintain component libraries, design tokens, and patterns
4. **Accessibility Testing Tools** (Axe, WAVE, Screen Reader Testing) - Audit WCAG compliance and validate with assistive technologies
5. **Handoff Platforms** (Figma dev mode, Zeplin, Handoff) - Transfer specifications to development with measurements and annotations
6. **Analytics & Heatmapping** (Hotjar, FullStory) - Understand user behavior and validate design effectiveness post-launch
7. **Design Critique & Feedback Tools** (Feedback, Frame.io) - Gather feedback from stakeholders and iterate quickly

## Design Integration Points

### Design Sends To:
- **PM Strategy** - Design research findings, competitive analysis, feasibility constraints on UX approach
- **Tech Architecture** - Interaction patterns, performance requirements, component structure for system design
- **PM Execution** - Design specs, annotation guides, edge case documentation for development tickets
- **QA** - Design specifications, state matrices, acceptance criteria for test case creation
- **Launch** - Design guidelines, component documentation, post-launch design support materials

### Design Receives From:
- **PM Strategy** - Business requirements, user personas, success metrics, competitive context
- **Tech Architecture** - Technical constraints, performance budgets, feasibility feedback on design approach

### Design Validation Gate
**Design Specification Approval** - Before handoff to development:
- User research validates core interactions
- Design spec includes all component states (default, hover, active, disabled, error, loading)
- Accessibility audit confirms WCAG AA compliance
- Developer handoff documentation is complete with measurements, tokens, and animation specs
- PM confirms design meets business requirements and success metrics

## Design's "Shift-Left" Role (Phase 1-2)

### Week 1-2: Problem Definition
- Participate in initial requirements workshop with PM and stakeholders
- Conduct user research (interviews, surveys) to validate problem assumptions
- Review competitive landscape and identify design opportunities
- Propose 2-3 design directions for PM and leadership consideration
- Share feasibility questions with Tech Architecture about interaction complexity

### Week 3-4: Concept & Exploration
- Create multiple concept mockups for user testing and stakeholder validation
- Work with PM to refine target user personas and success metrics
- Collaborate with Tech Architecture to understand technical constraints
- Design initial component structure and design system approach
- Identify accessibility requirements and design for WCAG AA from the start

**Shift-Left Principle:** Early design involvement surfaces user experience risks, technical feasibility questions, and QA test complexity before development begins. This prevents costly design changes mid-build and ensures everyone aligns on the user problem before code is written.

---

## Handoff Checklist

When design is complete, ensure:
- [ ] All user flows and interaction states are documented
- [ ] Design system components are defined with all states (default, hover, active, disabled, error, loading, empty)
- [ ] Accessibility audit completed and WCAG AA compliance confirmed
- [ ] Design tokens created (colors, typography, spacing, animations)
- [ ] Responsive behavior specified for mobile, tablet, desktop
- [ ] Animation and motion specifications documented (duration, easing, triggers)
- [ ] Performance loading states and skeleton screens designed
- [ ] Error states and empty states fully designed
- [ ] Design rationale documented for PM, Dev, and QA reference
- [ ] Figma/XD dev mode populated with measurements and annotations

## Success Metrics

- **User Satisfaction:** Post-launch NPS and user satisfaction scores
- **Task Completion Rate:** Users complete intended tasks without help
- **Accessibility Compliance:** Zero WCAG AA violations in accessibility audit
- **Time to Handoff:** Design specification complete within Phase 3-4 timeline
- **Design System Adoption:** 100% of components built to spec without deviation
- **Iteration Cycles:** Fewer design change requests during development (indicates clear specs)
